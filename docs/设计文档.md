## 二、核心模块设计

### 2.1 文档解析模块 (`parser/`)

#### 2.1.1 模块职责
- 接收上传的文件对象
- 根据文件扩展名调用对应的解析器
- 提取文档内的纯文本内容
- 保留基础段落结构

#### 2.1.2 类设计

```python
# parser/__init__.py
class BaseParser(ABC):
    """解析器基类"""
    @abstractmethod
    def parse(self, file_path: Path) -> List[str]:
        """解析文件并返回段落列表"""
        pass

class PDFParser(BaseParser):
    """PDF解析器"""
    def parse(self, file_path: Path) -> List[str]: ...

class DocxParser(BaseParser):
    """Word文档解析器"""
    def parse(self, file_path: Path) -> List[str]: ...

class MarkdownParser(BaseParser):
    """Markdown解析器"""
    def parse(self, file_path: Path) -> List[str]: ...

class TXTParser(BaseParser):
    """纯文本解析器"""
    def parse(self, file_path: Path) -> List[str]: ...

class ParserFactory:
    """解析器工厂"""
    @staticmethod
    def get_parser(file_ext: str) -> BaseParser: ...
```

#### 2.1.3 解析策略
| 格式 | 解析库 | 段落识别策略 |
|------|--------|-------------|
| PDF | PyPDF2 | 按\\n分割，过滤空行 |
| DOCX | python-docx | 按段落元素遍历 |
| MD | markdown+正则 | 按\\n\\n分割 |
| TXT | 内置open | 按\\n\\n分割 |

### 2.2 文本切片模块 (`chunker/`)

#### 2.2.1 模块职责
- 接收解析后的段落列表
- 按照「段落边界+字符长度」混合策略进行切片
- 每个切片绑定元数据（文档名、段落序号）

#### 2.2.2 类设计

```python
# chunker/__init__.py
@dataclass
class TextChunk:
    """文本切片数据结构"""
    content: str           # 切片原文
    doc_name: str          # 所属文档名称
    chunk_index: int       # 切片序号
    original_para_idx: int # 对应原文档段落序号

class TextChunker:
    """文本切片器"""

    def __init__(
        self,
        min_length: int = 200,
        max_length: int = 300,
        overlap: int = 50
    ):
        """
        参数:
            min_length: 单切片最小字符数
            max_length: 单切片最大字符数
            overlap: 相邻切片重叠字符数
        """

    def chunk(self, paragraphs: List[str], doc_name: str) -> List[TextChunk]:
        """
        执行切片
        - 优先按段落边界切分
        - 单段落过长时按字符长度强制截断（不截断完整句子）
        - 相邻切片保持一定重叠以保持语义连贯
        """
```

#### 2.2.3 切片算法流程

```
输入: paragraphs (段落列表), doc_name (文档名称)

1. 初始化结果列表 chunks = []
2. 初始化段落索引 para_idx = 0
3. 初始化缓存文本 current_text = ""

4. 遍历每个段落 para:
   4.1 如果 current_text + para 长度 < max_length:
       - 追加到 current_text
   4.2 否则:
       - 4.2.1 如果 current_text 长度 >= min_length:
         → 保存 current_text 为一个chunk
         → 取最后overlap字符作为下一个current_text开头
       - 4.2.2 处理当前段落:
         → 如果段落本身超过max_length，按句子边界截断
         → 剩余部分继续累积

5. 处理最后剩余的 current_text（如长度够min_length则保存）

输出: chunks (TextChunk列表)
```

#### 2.2.4 切片约束
- 单切片字符数：200-300字（目标280字）
- 严禁在句子中间强行截断
- 相邻切片保持50字重叠

## 三、大模型API接口设计

### 3.1 接口适配器模式

```python
# llm/__init__.py
class BaseLLM(ABC):
    """大模型接口基类"""

    @abstractmethod
    def generate(self, prompt: str, system_prompt: str = None) -> str:
        """生成回答"""
        pass

class OpenAICompatibleLLM(BaseLLM):
    """OpenAI兼容接口的大模型（如OpenAI、Azure、LocalAI等）"""

    def __init__(self, base_url: str, api_key: str, model: str, **kwargs):
        self.client = OpenAI(base_url=base_url, api_key=api_key)
        self.model = model

    def generate(self, prompt: str, system_prompt: str = None) -> str:
        messages = []
        if system_prompt:
            messages.append({"role": "system", "content": system_prompt})
        messages.append({"role": "user", "content": prompt})
        response = self.client.chat.completions.create(
            model=self.model,
            messages=messages
        )
        return response.choices[0].message.content

# 预留扩展: 豆包、通义千问等适配器
# class DoubaoLLM(BaseLLM): ...
# class QwenLLM(BaseLLM): ...
```

## 七、扩展性设计

### 7.1 大模型适配器扩展

新增大模型支持只需：
1. 在 `llm/` 下创建新的适配器类（如 `doubao_llm.py`）
2. 继承 `BaseLLM` 并实现 `generate` 方法
3. 在配置中添加对应配置项

### 7.2 文档解析器扩展

新增文档格式支持只需：
1. 在 `parser/` 下创建新的解析器类
2. 继承 `BaseParser` 并实现 `parse` 方法
3. 在 `ParserFactory.get_parser()` 中注册新解析器

### 7.3 切片策略扩展

当前采用固定字符长度策略，预留扩展点：
- 可配置不同文档类型使用不同策略
- 可扩展为语义分块（基于Embedding相似度）
- 可扩展为标题/段落结构感知分块
